# Governance authorization template

- **Status:** Authorization requested
- **Authorizing PR:**
- **Reviewed reference:** Commit SHA, merge SHA, ADR, PDR, or motion
- **Scope:**
- **Runtime / migration / permission / authority change:** None unless
  explicitly approved by the decision itself

## Decision outcome

| Field | Value |
|---|---|
| Decision | Pending |
| Aggregate | Pending required votes |
| Conditions acknowledged | Pending where applicable |
| Evidence | Direct GitHub comment or review links below |

## Motion

State the precise decision, explicit scope, conditions, exclusions, rollback,
and all required approver capacities.

| Approver | Capacity | Vote | Evidence |
|---|---|---|---|
| | | | |

## Vote template (reply on the authorizing PR)

~~~
APPROVE — <motion name>

Capacity: <Product governance | Engineering governance | Security advisory>
Reviewed reference: <commit SHA, merge SHA, ADR, PDR, or motion>
Vote: <Approve | Approve with conditions | Reject>
Conditions: <conditions or None>
~~~

The Evidence link must point to the genuine GitHub comment or review. GitHub's
system-generated `created_at` is the authoritative audit timestamp; do not add
a manual timestamp field to the vote or table.

## Preconditions and non-starts

List required conditions before any implementation or execution. Explicitly
list anything this motion does not authorize.

## Rollback

Describe how to reverse the authorized change. For a docs-only motion/template
change, restore the prior documentation in a follow-up docs-only PR; do not
alter historical evidence.
